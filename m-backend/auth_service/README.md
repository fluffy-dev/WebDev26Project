# Auth Service — Complete Service Documentation

## Purpose

Central identity provider for the entire platform. Responsible for:
- User registration and login
- JWT token issuance and refresh
- Token verification for Traefik ForwardAuth
- User profile retrieval
- Profile image management (admin-only uploads)

All other services trust identity headers injected by Traefik after this service validates a token.
No other service issues or validates JWTs.

---

## Technology Stack

| Concern            | Choice                          | Reason                                              |
|--------------------|---------------------------------|-----------------------------------------------------|
| Framework          | Django 5 + DRF 3.15             | Admin panel needed for ProfileImage; DRF for APIs   |
| Auth tokens        | djangorestframework-simplejwt   | Standard JWT; configurable lifetimes                |
| Database           | PostgreSQL 16                   | Primary persistence                                 |
| ORM                | Django ORM (psycopg3)           | Native async-ready driver                           |
| File storage       | django-storages + S3            | Profile images stored in S3, URL in DB              |
| Event publishing   | confluent-kafka                 | Publishes `user.registered` on successful signup    |
| Config             | python-decouple                 | `.env`-based config, no hardcoded values            |
| Testing            | pytest-django + factory-boy     | Unit + integration; factories over fixtures         |
| WSGI server        | Gunicorn                        | 4 workers, 60s timeout                              |

## Domain Model

```
UserEntity (frozen dataclass)
  id               : UUID
  username         : str
  email            : str
  password_hash    : str
  created_at       : datetime
  updated_at       : datetime
  profile_image    : ProfileImageEntity | None

ProfileImageEntity (frozen dataclass)
  id               : UUID
  image_url        : str
```

Domain entities are **pure Python**. Zero Django imports. They can be instantiated and tested
without a database or running server.

## Database Schema

```sql
-- Managed by Django migrations, generated with makemigrations
CREATE TABLE profile_images (
    id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    image      TEXT NOT NULL        -- S3 key, resolved to URL via django-storages
);

CREATE TABLE users (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    username      VARCHAR(150) UNIQUE NOT NULL,
    email         VARCHAR(254) UNIQUE NOT NULL,
    password      TEXT NOT NULL,     -- bcrypt hash via Django's make_password
    profile_id    UUID REFERENCES profile_images(id) ON DELETE SET NULL,
    is_active     BOOLEAN NOT NULL DEFAULT TRUE,
    is_staff      BOOLEAN NOT NULL DEFAULT FALSE,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Debezium monitors the `users` table via PostgreSQL logical replication (wal2json plugin).
Every `INSERT` produces an event on the `dbz.public.users` Kafka topic.

## API Endpoints

### `POST /login`

No auth required.

**Request**
```json
{ "login": "username_or_email", "password": "plaintextpassword" }
```

**Response `200`**
```json
{
  "user_id": "uuid",
  "access_token": "eyJ...",
  "refresh_token": "eyJ..."
}
```

**Errors**
- `401` — wrong credentials or user not found

---

### `POST /registration`

No auth required.

**Request**
```json
{
  "username": "alice",
  "email": "alice@example.com",
  "password": "min8chars",
  "profile_image": "uuid-or-null"
}
```

**Response `201`**
```json
{
  "id": "uuid",
  "username": "alice",
  "email": "alice@example.com",
  "created_at": "2024-01-01T00:00:00Z",
  "updated_at": "2024-01-01T00:00:00Z",
  "profile_image_url": "https://cdn.example.com/avatars/1.png"
}
```

**Errors**
- `409` — username or email already taken
- `404` — profile_image UUID does not exist

---

### `POST /refresh`

No auth required.

**Request**
```json
{ "refresh_token": "eyJ..." }
```

**Response `200`**
```json
{ "access_token": "eyJ..." }
```

**Errors**
- `401` — token invalid or expired

---

### `GET /users/{user_id}`

Requires `Authorization: Bearer <access_token>`.

**Response `200`**
```json
{
  "id": "uuid",
  "username": "alice",
  "email": "alice@example.com",
  "created_at": "...",
  "updated_at": "...",
  "profile_image_url": "https://..."
}
```

**Errors**
- `401` — missing or invalid token
- `404` — user not found

---

### `GET /verify`

**Internal only — called by Traefik ForwardAuth, not by clients.**

Validates the `Authorization: Bearer` header. On success returns:

```
HTTP 200
X-User-Id: <uuid>
X-Username: <username>
```

Traefik copies these headers into the forwarded request to downstream services.
On failure returns `401` and Traefik rejects the request before it reaches any service.

## Kafka Events Published

### Topic: `user.registered`

Published by `UserEventProducer` immediately after successful registration.
Also captured independently by Debezium CDC on the `users` table.

```json
{
  "event": "user.registered",
  "user_id": "uuid",
  "username": "alice"
}
```

**Message key**: `user_id` (enables partition locality for the same user's events)

**Consumer**: `balance_service` — creates an empty wallet on receipt.

---

## Inter-Service Communication

This service **produces** events. It does not consume any.
It does not make HTTP calls to any other service.

```
Client → Traefik → POST /login | /registration | /refresh   (no auth needed)
Client → Traefik → GET /verify                              (called by Traefik middleware)
Traefik ForwardAuth → GET /verify                           (internal, not client-facing)
auth_service → Kafka [user.registered]  → balance_service
Debezium → Kafka [dbz.public.users]     → balance_service  (CDC fallback / primary)
```

## Token Mechanics

| Token         | Lifetime   | Storage recommendation     |
|---------------|------------|----------------------------|
| Access token  | 5 minutes  | Memory (JS), do not persist|
| Refresh token | 7 days     | HttpOnly cookie or secure storage |

JWT payload for access token:
```json
{
  "user_id": "uuid",
  "exp": 1234567890,
  "iat": 1234567890,
  "token_type": "access"
}
```

## Profile Image Management

- Created exclusively via Django admin panel (`/admin`)
- On save: image file uploaded to S3 via `django-storages`
- Only the resulting S3 URL is stored in `profile_images.image`
- Users choose from existing images at registration time by UUID
- No file handling in the auth service itself at runtime

## Code Style & Principles

- **DDD layering**: `domain → application → infrastructure → presentation`.
  Inner layers never import from outer layers.
- **Dependency inversion**: use cases depend on abstract repositories, not ORM models.
- **KISS**: views do only three things — validate input, call use case, serialize output.
- **DRY**: `_STATUS_MAP` in exception handler maps all app exceptions in one place.
- **Google-style docstrings** on all public methods. No inline comments.
- **Type hints** on all function signatures.
- **Frozen dataclasses** for entities and DTOs — immutability by default.
- **Factory Boy** for test data — no raw `User.objects.create()` in tests.
- **Never** mock the ORM in integration tests — use real DB with `pytest.mark.django_db`.
- **Always** mock external deps (Kafka, S3) in unit tests.

## Debezium Connector Registration

Run once after `postgres_auth` and `debezium` containers are healthy:

```bash
curl -X POST http://localhost:8083/connectors \
  -H "Content-Type: application/json" \
  -d '{
    "name": "users-connector",
    "config": {
      "connector.class": "io.debezium.connector.postgresql.PostgresConnector",
      "database.hostname": "postgres_auth",
      "database.port": "5432",
      "database.user": "auth_user",
      "database.password": "auth_pass",
      "database.dbname": "auth_db",
      "database.server.name": "auth",
      "table.include.list": "public.users",
      "topic.prefix": "dbz",
      "plugin.name": "pgoutput"
    }
  }'
```

Produces events to topic: `dbz.public.users`
